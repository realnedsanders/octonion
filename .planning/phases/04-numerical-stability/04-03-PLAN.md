<objective>
Fix float32 overflow handling in analyze_stability.py so the script produces NaN-free JSON output and valid SC-4 mitigation ratios.

Purpose: UAT found 128 NaN values across the three JSON outputs (72 depth_sweep, 36 condition_numbers, 20 mitigation). Root cause: at deep layers (200-500), stripped chains produce f64 norms ~1e63. The f32 chain overflows to inf. Computing `(inf_value - huge_value).norm()` yields NaN. The script only guards against small f64 norms (`> 1e-30`) but never checks whether `out32` is finite. Additionally, `json.dump` with `default=float` emits bare `NaN`/`Infinity` literals, which is invalid JSON. Fixing these issues also unblocks SC-4 mitigation ratio assessment, since stable depth depends on non-NaN error data.

Output: Patched `scripts/analyze_stability.py` that runs to completion with zero NaN in output JSONs and valid SC-4 ratios.
</objective>

**Fix 1 — Stripped chain relative error (run_depth_sweep, ~line 262):**
After computing `out32` and `out64`, before the relative error calculation, add an isfinite check on `out32`. The current code:
```python
out64_norm = out64.norm()
if out64_norm.item() > 1e-30:
    rel_err = (out32.double() - out64).norm() / out64_norm
    errors.append(rel_err.item())
```
Change to:
```python
out64_norm = out64.norm()
if out64_norm.item() > 1e-30:
    if not torch.isfinite(out32).all():
        errors.append(float("inf"))
    else:
        rel_err = (out32.double() - out64).norm() / out64_norm
        errors.append(rel_err.item())
```
This ensures that when f32 overflows to inf, the error is recorded as inf (clearly diverged) rather than producing NaN via inf-minus-huge arithmetic. The `inf` will propagate correctly through `np.mean` (mean of list with inf = inf), accurately reflecting that the chain has diverged.

Also apply the same isfinite guard to the norm_ratios computation — if `out64` itself is non-finite (can happen at extreme depths), record `float("inf")` for norm_ratio.

**Fix 2 — Full network relative error (run_depth_sweep, ~line 343):**

out64_norm = out64.norm()

    rel_err = (out32.double() - out64).norm() / out64_norm
    errors.append(rel_err.item())

Add the same `torch.isfinite(out32).all()` check before the subtraction. Also add a check for `torch.isfinite(out64).all()` — at depth 500, even f64 can overflow for some algebras.

**Fix 5 — JSON serialization (save_json function, ~line 206):**
Replace the current `save_json` implementation:
```python
def save_json(data: dict, path: str) -> None:
    """Save data to JSON with float handling."""

```
With a version that replaces NaN and inf with JSON-safe values:
```python
def _sanitize_for_json(obj):
    """Recursively replace NaN/inf with null for valid JSON output."""
    if isinstance(obj, float):
        if not np.isfinite(obj):
            return None
        return obj
    if isinstance(obj, dict):
        return {k: _sanitize_for_json(v) for k, v in obj.items()}
    if isinstance(obj, list):
        return [_sanitize_for_json(v) for v in obj]
    return obj

def save_json(data: dict, path: str) -> None:
    """Save data to JSON, converting NaN/inf to null for valid JSON."""
    sanitized = _sanitize_for_json(data)
    with open(path, "w") as f:
        json.dump(sanitized, f, indent=2, default=float)
```
This produces valid JSON (no bare NaN/Infinity literals). Downstream consumers see `null` for diverged measurements, which is the JSON convention for "not available".
