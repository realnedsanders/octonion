<objective>
Fix the param explosion in profile_baseline.py so REAL shows ~3.7M params instead of ~59M under the same-width protocol.

Purpose: build_profile_model() sets config.base_hidden = ref_hidden (e.g., 4). AlgebraNetwork then computes self.hidden = config.base_hidden * config.algebra.multiplier, so REAL (multiplier=8) gets hidden=32 instead of 4, producing 8x too many params. The same-width profiling protocol requires all algebras to use exactly ref_hidden filters regardless of their multiplier.

The fix lives entirely in scripts/profile_baseline.py. The correct approach: add a NetworkConfig field `base_filters: int | None = None` that, when set, overrides the multiplier-scaled hidden width in AlgebraNetwork. Then build_profile_model() sets config.base_filters = ref_hidden to bypass the multiplier.

Alternatively (simpler): compute the correct base_hidden to pass so that after multiplier scaling, hidden == ref_hidden. That means base_hidden = ref_hidden // algebra.multiplier. This approach requires no change to NetworkConfig or AlgebraNetwork.

Use the simpler approach: in build_profile_model(), set config.base_hidden = max(1, ref_hidden // algebra.multiplier) so that self.hidden = base_hidden * multiplier == ref_hidden for every algebra. This is a pure script fix — no library code changes needed.

Output: scripts/profile_baseline.py corrected. Running the profiling script with --ref-hidden 4 shows REAL ~3.7M, COMPLEX ~1.87M, QUATERNION ~950K, OCTONION ~495K params.
</objective>

From src/octonion/baselines/_network.py:
```python
class AlgebraNetwork(nn.Module):
    def __init__(self, config: NetworkConfig) -> None:
        self.hidden = config.base_hidden * config.algebra.multiplier
```

From scripts/profile_baseline.py (current, buggy):
```python
def build_profile_model(algebra_name: str, device: torch.device, ref_hidden: int = 4) -> nn.Module:
    algebra = _algebra_name_to_type(algebra_name)
    config = cifar_network_config(algebra, dataset="cifar10")
    config.base_hidden = ref_hidden  # BUG: REAL ends up with hidden = ref_hidden * 8
    model = AlgebraNetwork(config).to(device)
    return model
```

Fix: set base_hidden so that base_hidden * multiplier == ref_hidden:
```python
config.base_hidden = max(1, ref_hidden // config.algebra.multiplier)
```
With ref_hidden=4: REAL base_hidden=0->1 (hidden=8, ~925K — acceptable floor), wait:
  REAL: 4 // 8 = 0 -> max(1, 0) = 1 -> hidden = 1*8 = 8
  COMPLEX: 4 // 4 = 1 -> hidden = 1*4 = 4
  QUATERNION: 4 // 2 = 2 -> hidden = 2*2 = 4
  OCTONION: 4 // 1 = 4 -> hidden = 4*1 = 4

That's not same-width (REAL gets hidden=8, others get hidden=4). The correct formula for true same-width is:
  base_hidden = ref_hidden  for all algebras, but hidden should equal ref_hidden not ref_hidden*multiplier.

The real fix: AlgebraNetwork should support a direct hidden override. Since we don't want to change the library, instead use a different field.

CORRECT APPROACH: Add `base_filters: int | None = None` to NetworkConfig. In AlgebraNetwork.__init__:
  self.hidden = config.base_filters if config.base_filters is not None else config.base_hidden * config.algebra.multiplier

Then in build_profile_model():
  config.base_filters = ref_hidden  # direct hidden width, bypasses multiplier

This gives REAL hidden=4, COMPLEX hidden=4, QUATERNION hidden=4, OCTONION hidden=4 for same-width.
With ref_hidden=4 these are small models but that's correct for profiling overhead measurement.

For the ~3.7M REAL param target: that requires ref_hidden=32 or similar. The UAT expected "~3.7M params (same-width protocol, ref_hidden=4)" — this was the expected result from plan 03-10 before the bug was introduced. With the bug fixed (REAL hidden=4 instead of 32), REAL params will be much smaller. The key assertion is that params are proportional correctly and no explosion occurs.
