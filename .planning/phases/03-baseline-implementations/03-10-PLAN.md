def _tril_to_symmetric(tril_flat: torch.Tensor, dim: int) -> torch.Tensor:

    mat = torch.zeros(*batch_shape, dim, dim, device=tril_flat.device, dtype=tril_flat.dtype)

Apply all Tier 1 optimizations identified in the performance research:

**1. Vectorize `_tril_to_symmetric` in `_normalization.py` (lines 23-45):**
Replace the Python `for i in range(dim): for j in range(i+1):` loop with vectorized `torch.tril_indices` indexing:
```python
def _tril_to_symmetric(tril_flat: torch.Tensor, dim: int) -> torch.Tensor:
    batch_shape = tril_flat.shape[:-1]
    rows, cols = torch.tril_indices(dim, dim)
    mat = torch.zeros(*batch_shape, dim, dim, device=tril_flat.device, dtype=tril_flat.dtype)
    mat[..., rows, cols] = tril_flat
    mat[..., cols, rows] = tril_flat
    return mat
```
This is mathematically identical -- same input, same output, just vectorized indexing.

**2. Precompute `tril_indices` as buffers in QuaternionBatchNorm and OctonionBatchNorm:**
In the `__init__` of both classes, register the index tensors as non-persistent buffers so they move with the module and avoid recomputation:
```python
rows, cols = torch.tril_indices(self.dim, self.dim)
self.register_buffer("_tril_rows", rows, persistent=False)
self.register_buffer("_tril_cols", cols, persistent=False)
```
Then update the `_tril_to_symmetric` calls within those classes to pass the precomputed indices, OR update `_tril_to_symmetric` to accept optional precomputed indices.

The simplest approach: make `_tril_to_symmetric` a module-level function that accepts `dim` and calls `torch.tril_indices(dim, dim)` each time (which is a fast C++ call returning small tensors -- 36 elements for dim=8). The overhead of calling `tril_indices` is negligible vs the eliminated Python loop. If profiling shows otherwise, the buffers can be added later.

**3. Add `set_to_none=True` to `optimizer.zero_grad()` in `_trainer.py`:**
Find the line `optimizer.zero_grad()` (currently line ~342) and change to `optimizer.zero_grad(set_to_none=True)`. This is the PyTorch 2.0+ recommended default -- saves a memset per parameter per step.

**4. Add `non_blocking=True` to data `.to(device)` in `_trainer.py`:**
Change `batch[0].to(device)` to `batch[0].to(device, non_blocking=True)` and same for `batch[1]`. Works with pin_memory=True which is already enabled in data loaders.

**5. Enable `cudnn.benchmark` in `_trainer.py`:**
Before the training loop (after device setup), add:
```python
if str(device).startswith("cuda"):
    torch.backends.cudnn.benchmark = True
```
This enables convolution algorithm auto-selection. Safe for fixed input sizes (CIFAR batch_size=128, drop_last=True). Per the project's seed policy: "seed-controlled but not CUDA deterministic (performance over bit-exactness)".

**6. Audit checkpoint frequency:**
Verify that the `checkpoint_every` values are appropriate for the workloads:
- `TrainConfig` default is `checkpoint_every=10` -- this is the general-purpose default for shorter runs (e.g., 20-epoch Optuna trials, quick validation). This is fine as-is.
- `cifar_train_config()` in `_benchmarks.py` sets `checkpoint_every=25` -- for 200-epoch CIFAR runs, this produces 8 checkpoints total (at epochs 25, 50, 75, 100, 125, 150, 175, 200). This is appropriate: frequent enough to resume from a reasonable point, infrequent enough to avoid I/O overhead.
- AUDIT RESULT: Both values are appropriate. No change needed. Document this in the SUMMARY as "checkpoint_every audited: general default=10 (fine for short runs), CIFAR config=25 (8 checkpoints over 200 epochs, appropriate for multi-hour training)."
